 Layer 1: Campaign Command Ingestion Service
🎯 Core Purpose
Layer 1 is the front door of the entire system. It receives commands from Airflow (or any other scheduler), validates them with Django, and publishes them to Kafka for downstream processing.

Think of it as a restaurant host:

Takes your order (START/STOP/PAUSE/RESUME)

Checks if it's valid (table exists, kitchen is ready)

Sends order to kitchen (Kafka) and gives you a receipt (202 Accepted)

📊 What It Actually Does
1. Receives HTTP Commands
REST API endpoints: /campaign/start, /stop, /pause, /resume, /complete

Accepts JSON payload with campaign_id, optional user_id, reason

Returns immediate 202 Accepted with tracking ID

2. Validates with Django API
For every command, it checks:

Exists: Does campaign 123 exist in Django?

Content (START only): Does it have message templates?

Audience (START only): Does it have valid recipients?

3. Publishes to Kafka
Creates standardized command message with command_id, correlation_id

Publishes to sms-commands topic

Uses Kafka for reliable, ordered delivery

4. Manages Authentication
JWT tokens for Django API

Automatic token refresh before expiry

Thread-safe token management

5. Tracks Requests with Correlation IDs
Every request gets unique correlation_id

Propagated to all downstream services

Enables end-to-end tracing

async def validate_for_action(campaign_id, action):
    # Check existence (always)

🔄 Message Lifecycle
Stage 1: HTTP Request
text
POST /campaign/start
Headers: X-Correlation-ID: corr_123
Body: {"campaign_id": 456}
Stage 2: Validation
text
→ GET /api/campaigns/456/ (Django)
← 200 OK with campaign data
→ GET /api/campaigns/456/message-content/
← 200 OK with templates
→ GET /api/campaigns/456/audience/
← 200 OK with recipient count
Stage 3: Kafka Publication
text
→ Produce to sms-commands
  Key: 456 (campaign_id)
  Value: {command message}
← Ack from Kafka
Stage 4: Response
text
← 202 Accepted
  Body: {command_id, timestamp}
📈 Performance Characteristics
Throughput
Designed for hundreds of commands per second

Each validation: 3-4 API calls to Django

Kafka production: < 10ms per message

Total latency: 200-500ms per command

Scaling
Stateless - can run multiple instances

Load balancer distributes HTTP requests

Kafka handles consumer groups automatically

🚨 Failure Scenarios
Django Unavailable
text
Problem: Can't validate campaigns
Action: Return 503 Service Unavailable
Impact: No new commands processed
Recovery: Retry with exponential backoff
Kafka Unavailable
text
Problem: Can't publish commands
Action: Fail the request (no queueing)
Impact: Commands lost if not accepted
Recovery: Wait for Kafka, then resume
Token Expired
text
Problem: JWT token invalid
Action: Refresh automatically
Impact: Single request may fail
Recovery: Retry with new token
