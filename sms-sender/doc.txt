Core Purpose
Layer 3 is the execution engine of the SMS campaign system. It takes ready-to-send messages from Redis, fires them at the SMS provider at massive scale (4000 TPS), and reports what happened - all without ever touching the database.

Think of it as a high-speed conveyor belt:

Input: Messages waiting in Redis (from Layer 2)

Process: Fire them at the provider at exactly 4000 per second

Output: Status events to Kafka (for Layer 4 to process)

📊 What It Actually Does
1. Consumes Messages from Redis
Pops messages in batches (default 1000) from sms:dispatch:queue

Each message contains: message_id, campaign_id, phone number, text, language, retry_count

Uses Redis pipelining for efficiency

2. Rate-Limited Sending
Maintains exactly 4000 transactions per second to the provider

Never exceeds this limit - providers will ban you if you do

Uses token bucket algorithm for smooth, burstable rate limiting

3. Sends to 3rd Party Provider
Makes parallel HTTP requests to the SMS provider's API

Uses connection pooling (200 concurrent connections)

HTTP/2 enabled for multiplexing

Each request includes our message_id for tracking

4. Handles Responses Intelligently
Outcome	Action
✅ Success	Publish SENT event to Kafka
🔄 Temporary Failure (429, 5xx, timeout)	Push to retry queue with delay
❌ Permanent Failure (400, 404, etc.)	Publish FAILED event to Kafka
5. Manages Retries
Exponential backoff: 60s → 300s → 900s

Uses Redis SETEX for delayed retry queue

Maximum 3 retries per message

Retry count preserved across attempts

6. Receives Delivery Reports
Webhook endpoint at /webhook/delivery

Provider sends delivery receipts (DELIVERED, FAILED, etc.)

Publishes to sms-delivery-status Kafka topic

7. Publishes Everything to Kafka
sms-send-status: Result of sending attempt (SENT/FAILED)

sms-delivery-status: Final delivery outcome from provider

Zero direct database updates - fully event-driven

The Token Bucket Algorithm
The rate limiter ensures exactly 4000 TPS:

python
# Simplified example
class TokenBucket:
    def __init__(self, rate=4000):
        self.rate = rate  # tokens per second
        self.tokens = rate
        self.last_refill = time.time()
    
    async def acquire(self, tokens=1000):
        now = time.time()
        elapsed = now - self.last_refill
        
        # Add new tokens based on elapsed time
        self.tokens = min(self.rate, self.tokens + elapsed * self.rate)
        self.last_refill = now
        
        if self.tokens >= tokens:
            self.tokens -= tokens
            return 0  # No wait
        else:
            # Need to wait for more tokens
            return (tokens - self.tokens) / self.rate

     └────────────┼──────────────┘
                  ▼
         ┌─────────────────┐
         │  Connection     │
         │    Pool         │

Each worker:

Waits for rate limit tokens

Pops batch from Redis

Sends batch in parallel (using asyncio.gather)

Processes responses

Loops forever

Retry Mechanism
Failed messages go through a sophisticated retry system:

text
Message fails (retryable)
         │
         ▼
┌─────────────────┐
│ Increment retry │
│ count to 1      │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ Push to Redis   │
│ with TTL=60s    │
│ key: sms:retry:msg_id
└────────┬────────┘
         │ After 60s
         ▼
┌─────────────────┐
│ Message expires │
│ and is lost?    │── NO! We need a retry scanner
└─────────────────┘
Note: Redis SETEX just stores the message with a TTL. When it expires, it's gone. A separate process (Layer 4) scans for expired retry messages and pushes them back to the main queue.

Kafka-First Design
This is the most important architectural decision:

text
Traditional (Bad):
Layer 3 ──HTTP──▶ Django ──DB──▶ Update campaign status
         (Tightly coupled, slow, blocking)

Event-Driven (Good):
Layer 3 ──Kafka──▶ sms-send-status topic
         │
         └──Kafka──▶ sms-delivery-status topic
         
Then later:
Layer 4 consumes these topics and updates Django
Why this matters:

Decoupling: Layer 3 doesn't care if Django is up or down

Scalability: Kafka can handle millions of events

Reliability: If something crashes, events are still in Kafka

Audit trail: Complete history of every message

📈 Performance Characteristics
Throughput Calculations
text
Target: 4000 TPS
Batch size: 1000 messages
→ 4 batches per second
→ 250ms per batch

With 4 workers:
→ Each worker processes 1 batch per second
→ 250ms for API call + processing
→ Comfortable margin
Connection Pool
text
200 connections ÷ 4 workers = 50 connections per worker
Each worker sends 1000 messages in parallel
→ 20 messages per connection
→ HTTP/2 multiplexing handles this efficiently
Memory Usage
Each message ~500 bytes

500,000 queue size = 250MB Redis memory

Retry queue additional ~50MB

Well within Redis limits

🚨 Failure Scenarios
Provider Outage
text
Problem: Provider API is down (5xx errors)
Detection: All requests failing with 5xx
Action: 
- Mark all as retryable
- Push to retry queue with delays
- Continue trying (provider may recover)
- Alert if >10% failure rate
Kafka Down
text
Problem: Can't publish status events
Detection: Kafka producer errors
Action:
- Log errors locally
- Continue sending? (No - we'd lose events)
- Better to pause until Kafka recovers
- Store failed events in memory with backpressure
Redis Down
text
Problem: Can't pop messages or store retries
Detection: Redis connection errors
Action:
- Pause all workers
- Health check fails
- Container restarts
Rate Limit Exceeded
text
Problem: Provider returns 429
Detection: Response status code
Action:
- Automatic retry with delay
- Adjust rate limiter? (Provider may have lowered limit)
- Alert if sustained 429s
